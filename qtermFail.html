TL;DR

We made the gateway send QTerm a byte-for-byte identical copy of QTerm's own Kermit Send-Init packet, and QTerm still rejected it with "Invalid packet type." That is proof the problem is not anything the gateway sends — qtermh1.com's Kermit receive parser is defective. It does not behave like its own published source (RECVK.Z / KUTIL.Z), and it will not accept even its own packet format. No change on the gateway side can fix it.

What works instead: XMODEM between QTerm and the gateway is rock-solid at the same 9600 8N1. Kermit works fine with other CP/M clients (kercpm3 in server mode) and with C-Kermit 10.0. Interactive Kermit remains available in the gateway's upload/download menus for those clients; QTerm users should use XMODEM (or a different QTerm build — see Conclusion).

Test rig

  • Gateway: Raspberry Pi 5 at 192.168.1.178, release build in ~/ethernetgateway/target/release/, verbose per-packet Kermit trace on the web /logs page (:8080).
  • Client: QTerm 4.3e, binary qtermh1.com, on an SC126 (Z80, CP/M / RomWBW). Source repo: ~/AltairRepos/qterm (also git.imzadi.de/acn/qterm). Key files: source/RECVK.Z (receive state machine), source/SENDK.Z (send), source/KUTIL.Z (rpack/spack/checksums), KERMIT.I.
  • Link: direct serial cable, SC126 UART ↔ Pi FTDI (/dev/ttyUSB1 = serial_b, modem-emulator mode, 9600 8N1, no flow control). "Modem mode" only emulates Hayes AT so QTerm can "dial"; there is no real modem.
  • Wire tap: socat -x -v interposed between the gateway and the FTDI, logging both directions to ~/serialtap.log. In captures below, > = gateway→SC126, < = SC126→gateway.

Symptom

Kermit transfers between the gateway and QTerm fail in both directions, with QTerm displaying "Invalid packet type." XMODEM works perfectly to the same machine at the same baud.

  • Download (gateway = Kermit sender): QTerm sends its initiating NAK, the gateway replies with the Send-Init (S), and QTerm rejects the S. Traces to RECVK.Z rinit → retab → dbadp (KUTIL.Z, string "Invalid packet type").
  • Upload (QTerm = Kermit sender): also "Invalid packet type", immediately, at SENDK.Z (sireta) → dbadp.
  • On the failing download, a stray ~ (tilde, byte 0x7e) is spilled to QTerm's terminal — a byte lifted straight out of our Send-Init (the REPT field). A healthy Kermit receiver never dumps packet bytes to the screen; a confused one does.

The journey

  1. Kermit server mode was fixed first (a different client). With kercpm3 on RomWBW, server-mode downloads worked but showed "2–3 retries" every time. Live captures root-caused two real gateway bugs — a case-sensitive GET lookup, and an unsolicited Send-Init that crossed the receiver's initiating NAK and got resent as a duplicate — both fixed (retries → 1, the single unavoidable receiver-init NAK). Those fixes shipped. See Gateway changes.
  2. Kermit was re-enabled in the interactive upload/download menus (it had been hidden while unreliable), now that the send path was solid for the clients that work.
  3. Interactive Kermit download to QTerm. The wait-for-receiver logic worked exactly as designed: QTerm sent its initiating NAK, the gateway consumed it and sent one Send-Init. QTerm rejected the S: "invalid packet type."
  4. Wire tap. Confirmed our Send-Init is byte-perfect on the wire and correctly checksummed, and that the gateway does not echo/reflect QTerm's NAK in data mode.
  5. Source analysis of rpack/rinit. Showed that with our exact bytes, QTerm's own source must parse the packet as type 'S' and accept it. The binary disagrees. (Details below.)
  6. QTerm-compat experiment. Emitted a Send-Init byte-structurally identical to QTerm's own — the full 13-field layout, QBIN='&', CHKT='3'. Still rejected.
  7. Trailing-XON experiment. QTerm terminates every packet with CR + XON (0x11); we did not. Appended the trailing XON so our Send-Init became 100% byte-identical to QTerm's own, all 20 bytes. Still rejected.

Wire captures (ground truth)

QTerm's own Send-Init (captured when QTerm was the sender) — the reference

01 30 20 53  7a 2c 20 40 2d 23 26 33 7e 20 20 20 20  31  0d 11
SOH LEN SEQ 'S'  MAXL TIME NPD PAD EOL QC QB CK RP CAP W  X1 X2  CHK  CR XON

LEN 0x30 = tochar(16) → 13 data bytes. MAXL=90, TIME=12, NPAD=0, PADC=NUL, EOL=CR, QCTL=#, QBIN=&, CHKT=3, REPT=~, then CAPAS/WINDO/MAXLX all zero. Every QTerm packet ends CR + 0x11 (XON) — its fixed trailer.

Why "invalid packet type" cannot come from our bytes

Reading QTerm's own source, the rinit (Send-Init-wait) state does this after rpack returns the packet type in A:

call  rpack        ; A = packet type, or 0 on timeout/bad-checksum
cp    'S'          ; type 'S'?  -> accept, ACK, move to file state
jr    nz,noris
...
noris: or a        ; A == 0 (timeout / checksum fail)?  -> send NAK
       jr nz,norif
       ...
norif: cp 'E'      ; type 'E' (error packet)?  -> handle error
       jr nz,retab
retab: call dbadp  ; else -> "Invalid packet type"

So dbadp fires only when rpack returns a checksum-valid packet whose type is neither 'S', 'E', nor 0. Two facts follow:

  • A garbled or wrong-length Send-Init fails the checksum → rpack returns 0 → QTerm NAKs (not "invalid packet type"). So our packet is not simply corrupted.
  • rpack reads the type byte at a fixed offset (3rd char after SOH), independent of the length field. Our type byte is 0x53 = 'S'. The data-length formula (E = LEN − cktyp − 0xF2) correctly yields our data count when the check type is '1' — which it must be, since the Send-Init bootstrap always uses a type-1 checksum.

Therefore, per the published source, our Send-Init must parse as 'S' and be accepted. The running qtermh1.com binary does something different — and the byte-identical test below proves the divergence is in the binary, not in us.

The decisive test

We eliminated every wire-level difference in turn until our Send-Init was a perfect copy of QTerm's own — same length, same 13 fields, same QBIN='&', same CHKT='3', same checksum '1', and finally the same trailing XON. All 20 bytes identical. QTerm rejected it exactly as before.

A Kermit receiver that rejects a byte-for-byte copy of its own Send-Init is broken. There is no packet we can construct that QTerm will accept, because we already sent it the one it emits itself.

The stray ~ spilled to the terminal on failure is consistent: 0x7e is a byte inside that Send-Init (the REPT field), and QTerm's receiver is dumping packet bytes to the display instead of consuming them — a confused parser, not a clean rejection.

Conclusion & recommendations

qtermh1.com's Kermit receive path is defective. It rejects even its own Send-Init format, so it cannot be made to work by any change on the gateway side.

Options, in order of value:

  1. Use a different QTerm build. The QTerm repo ships other binaries — qtermh2.com, qterm84.com, qterm82.com, qtermc3.com. If any has a working Kermit receiver, interactive Kermit will just work with no gateway change. (Untested as of this report — the recommended next step.)
  2. Use XMODEM for QTerm. Proven reliable at 9600 8N1, both directions.
  3. Disassemble qtermh1.com's rpack to pinpoint the defect — rigorous but slow Z80 reverse-engineering, and even if found we cannot patch the binary.

What does work (verified)

  • ✓ XMODEM — QTerm ↔ gateway, 9600 8N1, both directions.
  • ✓ Kermit server mode with kercpm3 (RomWBW CP/M) — clean downloads (one canonical receiver-init NAK, which the client tallies as "1 retry").
  • ✓ Kermit with C-Kermit 10.0 — full interop, verified by the test suite.
  • ✗ Kermit with qtermh1.com — both directions, "invalid packet type" (this report).

Gateway changes that came out of this investigation

Shipped (real fixes, kept)

  • kermit_wait_for_receiver (default on) — wait for the receiver's initiating NAK before sending the Send-Init, on both interactive downloads and server GET responses. Canonical receiver-driven start (da Cruz §4); avoids crossing the poke and sending a duplicate S.
  • Case-insensitive server GET — a CP/M client that uppercases a filename no longer fails "File not found" against a lower-case on-disk file.
  • Kermit re-enabled in the interactive upload/download menus — for the clients that work (C-Kermit, kercpm3, modern terminals).

Reverted (did not help)

  • kermit_qterm_compat — a QTerm-shaped Send-Init (full 13-field layout, QBIN='&', CHKT='3', trailing XON). Since a byte-identical Send-Init is still rejected, this shim provided no benefit and was removed to keep the code clean. This report is the record of that experiment.
